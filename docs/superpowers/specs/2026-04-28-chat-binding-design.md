# 飞书群目录绑定设计

**目标**

为飞书群提供可持久化的工作目录绑定，并基于 `open_id` 做全局访问控制。未绑定目录的群不处理普通请求，只提示先绑定。第一版不处理 `lark-cli` 稳定性层和全局 prompt / skills 注入。

**范围**

- 持久化 `chat_id -> cwd` 绑定关系
- 通过配置的 `open_id` 白名单控制可用用户
- 允许授权用户通过 `/cd` 建立或修改群目录绑定
- 未绑定群拒绝普通对话

**不在范围内**

- `lark-cli` 使用策略和预检查
- 全局 prompt / skills 控制点
- 通过聊天指令管理 `open_id` 白名单
- provider / session 的持久化

## 设计

### 1. 目录绑定存储

新增本地 JSON 存储文件，保存每个群的目录绑定。推荐默认位置：

- `data/chat-bindings.json`

结构使用对象映射，便于人工查看和编辑：

```json
{
  "version": 1,
  "bindings": {
    "oc_xxx": {
      "cwd": "C:\\work\\repo-a",
      "updatedAt": "2026-04-28T10:00:00.000Z"
    }
  }
}
```

### 2. 权限模型

新增全局 `open_id` 白名单配置，来源于环境变量。第一版不做分群权限差异，也不通过指令动态维护。

- 环境变量：`FEISHU_ALLOWED_OPEN_IDS`
- 格式：逗号分隔

只有白名单内用户可以：

- 使用普通对话
- 使用 `/cd` 绑定或修改目录
- 使用 `/resume`、`/agent` 等控制命令

### 3. 运行时行为

- 未授权 `open_id`：直接拒绝
- 已授权但群未绑定目录：
  - 普通消息：拒绝，并提示先用 `/cd <路径>` 绑定
  - `/cd`、`/help`：允许
- 已授权且群已绑定目录：
  - 沿用当前逻辑处理消息
  - `ChatManager` 优先使用持久化绑定目录作为该群当前目录

### 4. 命令语义

- `/cd <路径>`：
  - 校验目录存在
  - 更新运行时当前目录
  - 同时持久化为该群的新绑定目录
- `/cd`：
  - 仍显示最近目录菜单
  - 选择后同样更新持久化绑定

### 5. 测试策略

新增纯逻辑测试，覆盖：

- 绑定存储的落盘和重载
- `ChatManager` 从持久化绑定恢复当前目录
- `/cd` 更新绑定后重启仍可读取
- 未授权用户拒绝
- 未绑定群拒绝普通消息，但允许授权用户执行 `/cd`

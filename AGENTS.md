# Agent Guide

## 目的

本文件是 agent 在本仓库中工作的入口。

它定义：

- 仓库的事实来源
- 持久化任务状态的位置
- 当前 WIP 规则
- 完成前必须执行的验证
- 文档与提交要求
- 本仓库的项目专属约束

## 事实来源

仓库是当前项目的事实来源。

agent 应依赖本仓库中被跟踪或明确维护的文件。外部笔记、旧导出、临时文件、仓库外引用都不是权威来源，除非本文件明确声明。

当前有效实现位于仓库根目录。Python 历史实现已删除，不再作为新开发、验证或对照的直接依据。

需要参考 Python 迁移经验时，优先阅读 `docs/PYTHON_TYPESCRIPT_RUNTIME_BOUNDARY_NOTES.md` 和 `docs/task-archive/` 中的归档记录；不要引用已删除的 `apps/python/` 路径作为当前事实。

## 文档地图

- `AGENTS.md`：agent 规则、边界、工作流程和当前运行模型。
- `docs/PROGRESS.md`：当前项目状态、当前任务、有效基线、验证入口和下一任务。
- `docs/DECISIONS.md`：影响后续工作的长期决策，包含简短原因或依据，但不承载完整任务证据。
- `docs/task-archive/`：历史任务记录，包括 scope、验证、evidence、risk 和 commit 对应关系。
- `docs/TODO.md`：现有任务池和待整理问题清单。它不是唯一进度来源；当前活动任务以 `docs/PROGRESS.md` 为准。

当某类上下文会反复使用、且会让 `AGENTS.md` 过大时，在 `docs/` 下新增领域文档。

## 当前项目方向

- 保留并继续演进 TypeScript 实现。
- 当前项目结构已收敛为 TypeScript 单实现；仓库根目录即应用根目录。
- 不重新引入旧的多应用子目录作为应用根目录，除非用户明确要求进行新的结构迁移。
- 不把结构迁移、业务修复、测试脚本修复混入同一个任务。

## WIP 规则

遵守 WIP=1。

同一时间只允许一个任务处于 `in_progress`。在当前任务进入以下任一状态前，不开始新的独立任务：

- 已验证
- 明确阻塞
- 被用户取消

当前活动任务记录在 `docs/PROGRESS.md`。

## 会话开始

修改前：

1. 阅读 `AGENTS.md`。
2. 阅读 `docs/PROGRESS.md`。
3. 检查工作区状态。
4. 识别当前任务，或创建一个新任务。
5. 更新 `docs/PROGRESS.md`，确保只有一个当前 `in_progress` 任务。
6. 明确当前任务的验证命令。

## 修改纪律

- 修改范围限定在当前任务内。
- 优先采用能满足任务的最小改动。
- 不把无关清理、格式化、生成文件或顺手重构混入当前任务。
- 保留与当前任务无关的用户或工具已有改动。
- 如果发现独立问题，记录为后续任务，不静默扩大范围。

## 验证

任务必须通过仓库定义的验证命令后，才算完成。

默认验证命令记录在 `docs/PROGRESS.md`。

当前 TypeScript 主线的默认验证入口是：

```powershell
Set-Location C:\work\cc-feishu
npm run build
```

测试脚本当前存在已知不一致：部分 `package.json` 脚本依赖 `ts-node`，但依赖声明尚未收敛。不要在未修复前把全部测试脚本视为可靠验证矩阵。

验证后，在任务记录中写明：

- 使用的命令
- 验证结果
- 相关证据
- 剩余风险或已知缺口

如果验证失败，记录失败信息，并将任务标记为阻塞或仍在进行中。

## 会话结束

结束工作前：

1. 运行当前任务的验证命令。
2. 在 `docs/task-archive/` 中新增或更新任务记录。
3. 更新 `docs/PROGRESS.md`，记录当前任务状态、基线、观察项和下一任务。
4. 如果本次工作产生长期决策，更新 `docs/DECISIONS.md`，保留足够解释决策的原因；完整证据写入任务归档或领域文档。
5. 保持工作区可理解：没有无关临时文件、生成噪音或未解释改动。

## 提交约束

- 提交消息使用 Conventional Commits。
- TypeScript 改动使用 `<type>: [ts] <summary>`。
- 文档、仓库配置、CI 等非 Python/TypeScript 改动不加 `[py]` 或 `[ts]`。
- 每个 commit 只包含一个已验证工作单元。
- 不要包含无关文件、顺手格式化、生成噪音或其他任务的改动。
- 如果确实需要提交未验证 checkpoint，必须在 commit message 中标明这是 checkpoint，并说明尚未验证的内容。

## 编码约束

- 所有源码、脚本、文档文件统一使用 UTF-8 编码保存。
- 只要文件里包含中文，就不能使用 GBK、ANSI、Windows-1252 等本地编码保存。
- 特别是 TypeScript 源文件，若编码错误，`tsc` 会按 UTF-8 读取并把中文编进产物，最终在飞书里显示为乱码。

- 使用 PowerShell 写回文件时，不要依赖默认编码。
